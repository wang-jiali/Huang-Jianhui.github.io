组件设计

2018-09-14
阅读次数 0

文章目录

  1. 现状
  2. 优势
  3. 劣势
  4. 分析
  5. 总结

现状

    很多大大小小的公司，都有自己一套组件库。利用已封装好的组件库，在项目上做到模块的快速复用。大一点说就是在同一个公司里的产品，表现形式与交互能尽可能的保持统一。

    即使公司里没有这样一套组件库，也会选择一套较稳定、方便、好看的开源组件库，作为主要组件库进行开发。网上第三方开源组件库也非常多。各个技术栈的版本都有。像elementUI、iView、jqueryUI、weui等等。

优势

    无论是自己实现的一套组件库，亦或者是第三方的组件库。他们都有些许共性。

  • 每个组件库有自己独特的表现风格与交互方式。尽可能保持一个系统风格一致。
  • 可快速进行项目开发。无需过多考虑组件兼容问题。并且使用门槛低。

    当然，理想总是过于丰满的。一套组件库，真的能应付得了这么多平行的项目吗。除非你组件库足够强大，强大到把什么场景的应用都考虑进去了。这又回到了那个问题，是小而精悍好，还是大而齐全好，这种先有鸡还是先有蛋的哲学问题上。

劣势

  • 当组件库无法满足功能时，拓展相当困难。
  • 组件库无法满足设计师的设计时，要写一大堆样式层叠。难度不小于在做一个组件。
  • 过于庞大的api。（鬼能记得住这么多api，每个人都是最强大脑吗）
  • 当基于vue实现完一个组件库，产品老大突然说要用react。泪流满面。

    那这样说来，是不是就不用封装组件库了？话也不能这么说，组件库给项目开发带来的便利性，还是有目共睹的。特别是那种没有设计师参与的项目，选用一个
组件库撸起来是非常爽的。既然现有的组件库都存在这些问题。我们只能从组件设计谈起。

    一些基本原则就不多做累述，什么高内聚，低耦合，什么好莱坞原则。这类百度一下一大堆。

分析

    一个组件内应该包含哪些内容：

  • html、css，也就是UI层，表现层。
  • js，也就是行为。通过js可改变表现层。
  • 数据状态层。一个组件有多种状态，这些状态或是从应用场景读取，亦或是组件自身状态。
  • API。

    其中。我们将组件这些内容。进行分类，一类是组件表现形式，这类是多变的。不同的项目，有不同的设计师，不同的UI。一类是组件的核心逻辑、事件处理，这
类是比较稳定的。

    以modal对话框为例。尽管对话框可能存在千百种不同的表现形式，但是他们的核心逻辑以及事件处理是不变的。

    我们可以尝试着把UI层分离出去。不同的项目，去实现不同的UI，而核心逻辑与事件处理。可以预先写在组件内。这就是一个分离与半封装的思想，张鑫旭提出来的一种组件设计的思路。原文地址，这也是表现形式与行为更深层次的解读吧。这种实现，可能涉及到的一些维护成本以及项目周期。上述链接中已有回答。不在累述。

    其实还有另一种做法。感兴趣的不妨看一下。与张鑫旭的设计思想，有着异曲同工之妙。UI-MODEL

总结

    在已有并且稳定的组件库中，使用矛盾也日渐增强。有时候跳出这个圈子，从组件设计源头思考。不失为一种好的做法。
